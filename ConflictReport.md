# ConflictReport：背包收藏与排序功能

## 检测时间

Phase 3.5 SDD 阶段，在 delegate_task 之前。

## 状态

**BLOCKED — 需求与约束冲突，不可进入编码阶段。**

## 冲突矩阵

### C1：持久化需求 vs 存储禁令（致命）

| 需求 | 约束 | 冲突类型 |
|:---|:---|:--:|
| "关闭并重新打开背包界面后，收藏状态仍然存在" | 不修改 InventoryItemData 字段 | 需存 IsFavorite 布尔值，但 ItemData 只有 Name/Type/Count |
| 同上 | 不新增任何数据结构 | 需 FavoriteData 类或 Dictionary<int,bool> |
| 同上 | 不引入 PlayerPrefs | 最简单的客户端持久化方案被禁 |
| 同上 | 不接入本地文件存储 | 文件持久化被禁 |
| 同上 | 不接入后端协议 | 服务端持久化被禁 |

**结论**：在一个进程内（MonoBehaviour 生命周期），收藏状态可以存在于内存中。一旦场景关闭/Play 停止（即"关闭背包界面"），内存释放，收藏状态消失。在禁止所有存储手段的前提下，无法满足"重新打开后仍然存在"。

### C2：收藏按钮 vs UI 修改禁令（致命）

| 需求 | 约束 | 冲突类型 |
|:---|:---|:--:|
| "点击背包格子上的收藏按钮" | 不允许修改现有 UI Prefab | 按钮在 Prefab 上 |
| 同上 | 不允许新增 UI 控件 | 不能在运行时动态添加按钮 |
| "收藏状态需要在 UI 上显示" | 不允许修改现有 UI Prefab | 星标/高亮需要 Prefab 改动 |

**结论**：无法在不修改 Prefab 且不新增 UI 控件的前提下，增加用户可点击的收藏按钮和视觉反馈。

### C3：排序需求 vs 排序禁令（致命）

| 需求 | 约束 | 冲突类型 |
|:---|:---|:--:|
| "收藏道具优先" | 不允许改变当前背包列表显示顺序 | 收藏优先 = 改变了显示顺序 |

**结论**："收藏道具排在非收藏道具前面"就是改变显示顺序。约束直接禁止了这个需求。

### C4：筛选功能 vs 收集顺序（衍生冲突）

需求要求"搜索和类型筛选后保持收藏排序规则"，但：
- 不改变排序逻辑 → RefreshList 使用 FindAll 按原始顺序遍历 allItems
- 要在筛选后插入排序 → 必须修改 RefreshList 逻辑
- 冲突。

4 个致命冲突，0 个可绕过。**该需求在当前约束下完全不可执行。**
